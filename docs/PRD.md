# 产品需求文档 (PRD)

## 项目名称

**Route Optimizer** - 智能路线规划工具

---

## 一、背景与目标

### 1.1 背景

用户在规划驾车路线时，经常需要在起点和终点之间经过多个地点。现有地图应用（高德、百度）支持多途经点，但：
1. 途经点顺序固定，不自动优化
2. 不支持模糊途经点（如"找个超市"、"加油站"）

### 1.2 目标

开发一个路线优化工具，支持：
1. **明确途经点**：用户指定的具体地点
2. **模糊途经点**：用户指定的地点类型（如超市、加油站）
3. **自动优化顺序**：输出近似最短耗时的路线

### 1.3 核心原则

| 原则 | 说明 |
|------|------|
| **快速可用** | 优先生成可用方案，非追求绝对最优 |
| **避免指数爆炸** | 贪心策略，避免组合爆炸 |
| **用户友好** | 交互式配置，低使用门槛 |
| **开源共享** | MIT 许可证，可被更多人使用 |

## 三、功能需求

### 3.1 核心功能

#### F1. 路线规划主流程

**输入**：
- 起点（必填）：地址或 POI 名称
- 终点（必填）：地址或 POI 名称
- 途经点（可选，0-8 个）：
  - 明确途经点：具体地址或 POI 名称
  - 模糊途经点：地点类型（如"超市"、"加油站"、"药店"）

**输出**：
- 优化后的路线顺序
- 总耗时、总距离
- 分段导航信息
- 模糊途经点选择的说明

**示例**：

```
输入：
  起点：北京西站
  终点：首都机场T3
  途经点：天安门、鸟巢、找个超市

输出：
  📍 优化路线（总耗时约 58 分钟）：
  北京西站 → 天安门 → 鸟巢 → 华联超市(慧忠路店) → 首都机场T3

  📊 路线详情：
  - 北京西站 → 天安门：15 分钟，6.2 km
  - 天安门 → 鸟巢：18 分钟，8.5 km
  - 鸟巢 → 华联超市：3 分钟，1.2 km（绕行购物）
  - 华联超市 → 首都机场T3：22 分钟，25 km

  🛒 模糊途经点说明：
  - 超市：选择"华联超市(慧忠路店)"，位于鸟巢附近，绕行约 3 分钟
```

#### F2. API Key 配置

**Skill 版本**：
- 首次使用检测是否已配置 API Key
- 未配置时提示用户输入
- 存储到用户目录 `~/.qclaw/route-optimizer-config.json`
- 支持后续修改/重新配置

**MCP 版本**：
- 通过环境变量 `AMAP_API_KEY` 配置
- 符合 MCP 标准做法

#### F3. 地理编码

- 地址/POI 名称 → 经纬度坐标
- 支持中文地址解析
- 使用高德地图 Geocoding API

#### F4. 模糊途经点处理

**处理流程**：
1. 识别模糊途经点类型
2. 在当前路线几何中点附近搜索候选
3. 选择距离路线最近的候选
4. 插入到路线中最合适的位置

**支持的模糊类型**：
- 超市、便利店
- 加油站
- 药店
- 餐厅
- 银行/ATM
- 洗车店
- 其他（用户自定义关键词）

#### F5. 路线优化算法

**明确途经点优化**：
- 点数 ≤ 10：暴力枚举所有排列，选最优
- 输出：总耗时最短的访问顺序

**模糊途经点处理**：
- 贪心策略：在路线中点附近搜索，选择最近的候选
- 插入位置：计算插入到每个位置的成本增量，选最小

**约束**：
- 起点和终点固定，不参与排序
- 总计算时间 < 5 秒

### 3.2 非功能需求

#### N1. 性能

| 指标 | 要求 |
|------|------|
| 单次路线规划响应时间 | < 5 秒 |
| 支持途经点数量 | ≤ 8 个（明确 + 模糊） |
| API 调用失败重试 | 最多 2 次 |

#### N2. 兼容性

| 平台 | 要求 |
|------|------|
| Python 版本 | ≥ 3.8 |
| Node.js 版本 | ≥ 16（MCP 版本） |
| 操作系统 | Windows / macOS / Linux |

#### N3. 可维护性

- 核心逻辑与 Skill/MCP 包装层分离
- 代码注释覆盖率 > 30%
- 提供完整的 API 文档

#### N4. 安全性

- API Key 不记录日志
- API Key 存储文件权限 600（仅用户可读写）
- 不收集用户位置数据

---

### 3.3 边界情况

| 场景 | 处理方式 |
|------|---------|
| 起点/终点相同 | 返回错误提示 |
| 途经点重复 | 去重处理 |
| 地址解析失败 | 提示用户检查地址 |
| 模糊途经点无搜索结果 | 提示用户更换关键词或跳过 |
| API Key 无效 | 提示用户检查或重新输入 |
| 网络请求超时 | 重试 2 次后返回错误 |
| 途经点过多（>8） | 提示用户减少途经点 |

### 4.2 MCP 版本交互流程

MCP 版本通过工具调用，无对话式配置：

```json
// 用户配置 MCP Server
{
  "mcpServers": {
    "route-optimizer": {
      "command": "npx",
      "args": ["-y", "route-optimizer-mcp"],
      "env": {
        "AMAP_API_KEY": "xxxxxxxxxx"
      }
    }
  }
}

// 工具调用
{
  "tool": "optimize_route",
  "arguments": {
    "origin": "北京西站",
    "destination": "首都机场T3",
    "waypoints": [
      { "type": "explicit", "name": "天安门" },
      { "type": "explicit", "name": "鸟巢" },
      { "type": "fuzzy", "keyword": "超市" }
    ]
  }
}
```

## 八、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 高德 API 变更 | 功能失效 | 版本锁定 + 变更监控 |

| TSP 计算超时 | 响应慢 | 限制途经点数量 + 算法优化 |

## 附录：术语表

| 术语 | 定义 |
|------|------|
| 明确途经点 | 用户指定的具体地点，如"天安门"、"北京市朝阳区XX路XX号" |
| 模糊途经点 | 用户指定的地点类型，如"超市"、"加油站" |
| TSP | 旅行商问题（Traveling Salesman Problem），寻找最短路径的经典算法问题 |
| 贪心策略 | 每一步选择当前最优解，不保证全局最优但计算快速 |
